# Transactions

- A unit of work that is **all-or-nothing**: `BEGIN` ... `COMMIT` / `ROLLBACK`

## ACID

- `Atomicity` ‚Äî all statements apply, or none do. Delivered by the undo/rollback mechanism
- `Consistency` ‚Äî the DB moves from one valid state to another (constraints, FKs, triggers hold)
- `Isolation` ‚Äî concurrent transactions don't see each other's partial work. Delivered by locking and/or MVCC
- `Durability` ‚Äî once committed, it survives a crash. Delivered by the [WAL](09.wal.md)

## Read phenomena

- `Dirty read` ‚Äî reading another transaction's uncommitted change
- `Non-repeatable read` ‚Äî re-reading a **row** returns different data (someone updated it)
- `Phantom read` ‚Äî re-running a **query** returns different rows (someone inserted/deleted)
- `Lost update` ‚Äî two read-modify-writes race, one is silently overwritten

## Isolation levels

- The SQL standard defines them by which phenomena they *permit*:

| Level | Dirty read | Non-repeatable | Phantom |
|---|---|---|---|
| Read uncommitted | ‚úÖ | ‚úÖ | ‚úÖ |
| Read committed | ‚ùå | ‚úÖ | ‚úÖ |
| Repeatable read | ‚ùå | ‚ùå | ‚úÖ |
| Serializable | ‚ùå | ‚ùå | ‚ùå |

- Higher isolation = more correctness, less concurrency
- **The defaults differ per engine**, which is a classic portability bug:
  - PostgreSQL ‚Üí `read committed` (its "repeatable read" is actually snapshot isolation, and blocks phantoms too)
  - MySQL / InnoDB ‚Üí `repeatable read`
  - SQLite ‚Üí `serializable` (it has only one writer, so it gets this for free)
  - Oracle / SQL Server ‚Üí `read committed`
- Note the standard describes levels by *anomalies*, not by *implementation* ‚Äî so two engines can both claim "repeatable read" and still behave differently

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
```

## How isolation is implemented

- `Locking (pessimistic)` ‚Äî a reader/writer takes a lock and others wait. Simple, but readers block writers
- `MVCC (optimistic)` ‚Äî every write creates a **new version** of the row; readers see the snapshot valid at their start time and never block
  - Used by PostgreSQL, MySQL/InnoDB, Oracle
  - The cost is garbage: dead row versions must be cleaned up later (PG's `VACUUM`, InnoDB's purge thread)
  - SQLite does **not** do MVCC ‚Äî it locks the whole database file instead
- `SELECT ... FOR UPDATE` takes an explicit row lock inside an MVCC engine, the usual fix for lost updates

## Deadlocks

- Two transactions each hold a lock the other wants. The engine detects the cycle and **kills one** with an error
- Mitigation: always acquire locks in a consistent order; keep transactions short; retry the victim
- Application code that runs transactions **must be prepared to retry** ‚Äî this is not optional at serializable isolation

## Practical notes

- Keep transactions short ‚Äî a long-running one holds locks, pins old row versions, and blocks log recycling
- Never leave a transaction open across a network call or user think-time
- DDL is transactional in PostgreSQL and SQLite (you can roll back a `CREATE TABLE`), but **not** in MySQL ‚Äî it implicitly commits
